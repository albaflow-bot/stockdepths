# BinDesk Procedural Playbook

> 이 문서는 BinDesk 자체 개발 과정에서 확정된 결정사항과 검증된 절차다.
> 새 프로젝트 생성 시 자동 전파되며, Vibe Coder Claude 가 코딩 시 자동 참조한다.
> 사용자가 명시적으로 다르게 결정하지 않는 한, 모든 결정 시 이 playbook 을 따른다.


## 🔴 Critical — 반드시 따라야 함

### 실패 시 사용자에게 묻지 않는다 — 시스템이 진단한다

**핵심**: 자동 작업이 실패하면 'why did it fail?' 류의 모달·prompt 를 띄우지 말 것. 대신 system-activity 로 로깅하고 자동 재시도·대안 제시.

**언제 적용**: 자동 워크플로우의 실패 처리 / catch 절 / Promise.reject 핸들러

**이유**: 사용자가 직접 결정할 수 없는 내부 진단을 사용자에게 떠넘기는 게 가장 큰 UX 실수다 (Phase 31 메타세션 사용자 직접 지적: '네가 진단해서 자동으로 진행하고는 내게 뭐가 잘못됐냐고 물어보면 답변할 것이 없다').

**피할 것**: alert() / window.prompt() 로 사용자에게 진단 요청

---

### 보류·삭제·거부는 분리된 세 액션이다

**핵심**: 한 항목을 처리하는 데 '지금 진행' 외에 보류(나중에)·삭제(완전 제거)·거부(시스템이 안 함)의 세 갈래가 모두 필요. 한 가지 부정 액션으로 통합하면 안 됨.

**언제 적용**: 사용자가 처리할 수 없는 항목이 큐에 남아 무한 알림을 받는 상황을 막는다.

**이유**: 버그 수신함·작업 대기열·자동 점검 결과 항목 모든 곳에서 적용. 일괄 자동 수정은 '보류' 항목을 자동으로 제외해야 한다.

**피할 것**: '지금 처리 / 영구 삭제' 이분법만 제공

---

### 임계 자원은 in-memory + DB 락 이중 가드

**핵심**: 강제 종료·다중 인스턴스 시나리오에서 in-memory 락만으로는 불충분. 토큰·세션 같은 비싼 자원에는 DB 수준 락도 함께. 부팅 시 stale lock 자동 정리.

**언제 적용**: 한 가지 가드만 두면 force-quit 후 재실행 연타로 토큰을 중복 소비한다.

**이유**: 비싼 외부 호출 (Claude 디스패치, 빌드, 배포, 결제) 직전

**피할 것**: OnceLock<Mutex<HashSet>> 하나만으로 끝낸다

---

### Sane default + override 패턴

**핵심**: 설정 파일·토큰·캐시 등이 없으면 안전한 기본값을 적용. 절대 에러로 막지 말 것.

**언제 적용**: 첫 진입의 마찰을 0 으로. 사용자가 '왜 안 되지?' 하는 5초가 80% 이탈을 만든다.

**이유**: config 로드·design_tokens·feature flag 등 모든 부수적 자원

**피할 것**: "config.json not found, please create" 류 에러로 시작

---

### 비동기 작업 UI는 스피너 단독 금지

**핵심**: 경과 시간·예상 시간·진행률 바 셋 중 최소 둘 이상이 보여야 함. TimedProgress 컴포넌트 사용.

**언제 적용**: 스피너만 보이면 멈춘 것 같고, 사용자는 강제 종료 → 토큰 손실로 이어진다.

**이유**: 1초 이상 걸리는 모든 비동기 작업

**피할 것**: <Spinner /> 단독 사용

---

### BinDesk 개발 세션 자체가 절차 학습 데이터다

**핵심**: 이 앱을 만드는 과정에서 발견된 모든 결정·패턴·반패턴은 새 프로젝트에 자동 전파되어야 한다. 메타 학습 루프.

**언제 적용**: BinDesk 의 목표는 '수정 없는 완성품' 인데, 그 목표는 BinDesk 자체의 절차 품질에서 시작한다.

**이유**: 사용자가 BinDesk 를 직접 수정·확장하는 모든 세션 (현재 메모리에는 '북극성 = 수정 없는 완성품' 으로 기록)

## 🟡 Normal — 권장 기본

### 권장값은 항상 근거 데이터와 함께 제시

**핵심**: '권장: 2' 가 아니라 '권장: 2 (이 PC 코어 8개 기준)'. 머신·세션·이력 등 추천의 근거가 UI에 동봉되어야 한다.

**언제 적용**: Settings 슬라이더·기본 선택·자동 분류 추천 등 모든 추천 UI

**이유**: 권장 클릭으로 즉시 적용 가능. 사용자가 왜 그 값인지 추정하지 않아도 된다.

**피할 것**: '권장: 5' 같이 근거 없는 매직 넘버

---

### 실패율 누적 → 시스템 스스로 차단

**핵심**: 최근 N분 내 X건 실패 시 해당 자동 기능을 자동 비활성화. 사용자가 매번 멈추지 않아도 됨.

**언제 적용**: 사용자가 실패한 시스템을 계속 재시도해 토큰·시간·신뢰를 잃는 것을 막는다.

**이유**: 일괄 자동 수정·자동 디스패치·배포 자동화 등 반복 작업

**피할 것**: 실패해도 계속 시도, 사용자가 직접 멈춰야 함

---

### 긴 작업은 페이즈 경계에서만 보고

**핵심**: Phase 안에서는 묵묵히 진행, 페이즈 종료 시 한 화면 요약. 결정 필요시에만 1줄 질문.

**언제 적용**: 잦은 보고는 사용자 흐름을 끊고, 너무 늦은 보고는 의도 이탈을 누적시킨다. 페이즈가 자연 체크포인트.

**이유**: 5+ 페이즈에 걸친 대규모 변경, 다단계 워크플로우

**피할 것**: 매 작업마다 "진행해도 될까요?" 묻기

---

### 선택사항이지만 언제든 1-클릭 접근

**핵심**: 주 워크플로우를 방해하지 않으면서, 영구 진입점이 항상 보이는 위치에 있어야 함. 헤더 아이콘이 정석.

**언제 적용**: 메인 경로를 막지 않으면서 발견 가능성을 유지한다.

**이유**: Optional but valuable 한 기능 (디자인 라운드, 고급 설정, 통계 보기)

**피할 것**: 강제 모달로 진입 강요 / 깊은 메뉴 속에 숨김

---

### 동일 기능도 페르소나별 UI 분기

**핵심**: 디자인 감각 있음 vs 없음, 코딩 경험 많음 vs 적음 등 핵심 차이는 첫 화면에서 분기. '가이드 받기 / 직접 표현' 2-카드 패턴.

**언제 적용**: 표현을 강요하면 사용자가 멈춘다. 인식(클릭)으로 치환하면 진행한다.

**이유**: 사용자가 자신의 어휘로 표현할 수 없을 가능성이 있는 모든 화면

**피할 것**: 어휘 없는 사용자에게 자유 입력만 제공
